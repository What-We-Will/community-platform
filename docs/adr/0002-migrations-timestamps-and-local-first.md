# ADR-0002 — Timestamp-based migrations and local-first authoring

**Status:** Accepted 2026-05-19
**TL;DR:** New migrations are timestamp-named (`YYYYMMDDHHMMSS_<slug>.sql`) and authored against each contributor's own personal Supabase project. CI gates filename collisions.
**Author:** @tonyrosario
**Sponsoring Lead:** @timchaffee

## Context

The project tracks database schema in `supabase/migrations/` as sequentially-numbered SQL files (`001_*.sql` through `057_*.sql`). Sequential numbering produces two failure modes when contributors work in parallel:

1. **Filename collision at merge.** Two contributors independently pick the same next number; whichever PR merges first claims it, and the second must be renumbered before merge.
2. **Preview-DB drift before merge.** Each PR's preview build runs `supabase db push` against the shared preview DB. Once PR A's migration applies, every subsequent PR that lacks it fails on `db push` (`Remote migration versions not found in local migrations directory`) — preview is ahead of those branches, and they cannot move until A merges (or drops the migration) and they rebase past it. The result is a cascade of unrelated CI failures.

Infrastructure constraints shape the available solutions. The org runs on Supabase Pro for production and preview, but does not auto-provision per-developer or per-PR databases. Contributors today author migrations against their own personal Supabase free-tier project on a separate account — an isolated authoring surface at zero project cost. Other isolation strategies (Pro Branching on the org plan, CI-spun ephemeral databases, Docker Postgres locally) are evaluated in *Alternatives considered*.

GitHub Issue #78 surfaced the collision problem after two PRs raced on the same migration number. The discussion concluded that the right time to standardize is now, at ~5 active migration authors, before semantic conflicts (two PRs altering the same column) start appearing.

## Decision

We will adopt three changes immediately and defer a fourth with explicit revisit triggers:

1. **Adopted: timestamp-based filenames.** New migrations use `YYYYMMDDHHMMSS_<slug>.sql`, generated by `supabase migration new <slug>`. Migrations `001`–`057` are grandfathered and frozen; new timestamped files sort after them lexicographically.

2. **Adopted: local-first authoring.** Contributors author and test migrations against their own isolated Supabase project (a personal free-tier cloud project today; Docker via `supabase start` is a future option). Migrations only reach the shared preview DB via merged PRs on `main`.

3. **Adopted: CI filename-collision gate.** A bash script (`scripts/ci/check-migration-collisions.sh`) runs unconditionally on every preview build and fails any PR that adds a migration filename already present on `origin/main`.

4. **Deferred: Supabase Branching (Pro plan, ~$25/mo per project).** Branching gives each PR an ephemeral hosted database and is the preferred long-term answer at scale. We will revisit when pain accumulates — see "Revisit triggers" below. The free fallback is **CI ephemeral databases** (a Postgres container spun up per CI job, with the migration set replayed), which is more work to set up but sustainable at zero recurring cost.

## Alternatives considered

**Option 1 — Timestamp-based filenames.** Adopted. Removes the collision class entirely for new migrations. Cost: a permanent visual split between the legacy `0NN_*` range and the new `YYYYMMDDHHMMSS_*` range. Worth it.

**Option 2 — Local-first authoring.** Adopted. Each contributor maintains an isolated Supabase project — a personal free-tier cloud project today; Docker via `supabase start` is a future option that removes the dependency on Supabase availability. The shared preview DB stops being a contended authoring surface; it becomes a deployment target.

**Option 3 — Supabase Branching (Pro plan).** Deferred. Branching is the cleanest answer — every PR gets a real isolated database with the full migration set applied — but it costs ~$25/month, and at ~5 active migration authors the friction it removes is not yet worth the spend. We will reopen this option when one of the revisit triggers fires.

**Option 3-fallback — CI ephemeral databases.** Named as the sustainable free escalation if Option 3's recurring cost is not acceptable when the revisit point arrives. Each CI job provisions a fresh Postgres container, replays the migration set, and exercises the schema. More implementation work than Branching, but no recurring fee.

**Option 4 — CI filename-collision gate.** Adopted, scoped to filename-collision only. It does not detect semantic conflicts (two PRs `ALTER` the same column with incompatible intent); those require an actual database to apply against and are deliberately out of scope for this milestone.

### Revisit triggers for Option 3 / 3-fallback

Whichever fires first:

- ~5–7 active migration authors and contributors are reporting DX strain (slow author-test cycles on personal projects, repeated re-seeds, contention on the shared preview DB despite local-first authoring).
- The first **semantic conflict** appears (two PRs `ALTER` the same column or otherwise change the same schema object in incompatible ways).

**Evidence channel:** the `migration-pain-signal` GitHub issue label. Contributors file an issue with that label whenever they hit migration friction. Platform leads review the label quarterly. When either trigger above fires based on labeled evidence, we open a new ADR to choose between Option 3 and Option 3-fallback.

## Consequences

- Legacy `001`–`057` are grandfathered. The migrations directory will have a permanent visual split between the legacy range and the new timestamped range. This is cosmetic, not a correctness concern.
- Each contributor maintains a personal Supabase project for authoring (free-tier sign-up on their own account). Docker via `supabase start` is a documented future option but is not currently required. Documented in `README.md`.
- The CI collision gate runs on every PR, including PRs that touch no migrations. The overhead is ~1 second of `git diff` against `origin/main` — cheap enough to leave unconditional and avoid the "did the gate run?" failure mode.
- The `migration-pain-signal` label becomes the operational signal channel for re-opening Option 3 / 3-fallback. Platform leads must remember to review it quarterly; an unreviewed label drift is itself a signal.
- Pro Branching (~$25/mo) is the preferred answer at scale. CI ephemeral DBs is the free fallback if the recurring spend is not sustainable.
- This ADR does not change `production.yml` — the production deploy path does not push migrations today and remains out of scope.
- Failure mode #2 (preview-DB drift) is only partially mitigated by this ADR. Local-first authoring keeps contributors from manually pushing unmerged migrations to preview, but the preview-deploy CI step still applies each PR's migrations to preview before merge. Fully eliminating the cascade requires per-PR isolated databases (Option 3 or 3-fallback), which are deferred.

### Pain-point timeline

Indicative — used to anticipate when the deferred options will need to be reopened. Numbers reflect the project's own scale-up rhythm and are revisable.

- **~5 contributors (current):** naming collisions only. Handled by this ADR's adoption.
- **~10 contributors:** DX strain on the shared preview DB starts to bite — contention on test data, slow re-seed cycles. Local-first authoring still sufficient; the shared preview DB feels noisier.
- **~20 contributors:** semantic conflicts begin appearing (two PRs alter the same table). M15's scope no longer covers; Option 3 or 3-fallback becomes needed.
- **50+ contributors:** one of Pro Branching or CI ephemeral DBs is mandatory. Local-first alone produces post-merge surprises that erode trust in `main`.
